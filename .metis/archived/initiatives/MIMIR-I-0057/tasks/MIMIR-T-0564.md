# Migration: Add homebrew_monster_id to module_monsters

## Parent Initiative

[[MIMIR-I-0057]]

## Objective

Create a new Diesel migration that adds `homebrew_monster_id` to the `module_monsters` table, makes `monster_name`/`monster_source` nullable, adds a CHECK constraint ensuring exactly one reference type, and updates the Diesel schema + Rust models accordingly.

- [ ] `monster_name` and `monster_source` columns become nullable (SQLite requires table rebuild)

- [ ] Existing tests pass (no regressions from nullable change)
- [ ] New unit test verifies the model builder prevents setting both catalog and homebrew refs

## Implementation Notes

### Technical Approach
SQLite doesn't support `ALTER COLUMN` to make columns nullable. The migration must:
1. Create new table with desired schema
2. Copy data from old table
3. Drop old table
4. Rename new table

This is the same pattern used in other Mimir migrations (see `migrations/` for examples).

The `module_monsters` table is referenced by `token_placements(module_monster_id)`, so the rebuild must preserve FKs or temporarily disable them.

### Key Files
- `migrations/NNN_homebrew_module_monsters/up.sql` — new migration
- `migrations/NNN_homebrew_module_monsters/down.sql` — reversal
- `crates/mimir-core/src/schema.rs` — auto-generated by Diesel
- `crates/mimir-core/src/models/campaign.rs` — `ModuleMonster`, `NewModuleMonster`

### Dependencies
None — this is the foundation task.

### Risk Considerations
- Table rebuild with FK references from `token_placements` needs careful handling
- Must test with existing data to ensure no breakage on migration
